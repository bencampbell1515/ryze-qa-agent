# QA Report Redesign — HTML + PDF

**Date:** 2026-05-06
**Status:** Approved for implementation

---

## Goal

Replace the `.docx` report with an HTML-primary report that is visually polished, usable without any app, and includes cropped screenshots and plain-English summaries. A clean PDF export of the severity view is generated alongside it for formal sharing.

---

## Output Files

| File | Purpose |
|---|---|
| `output/audit-report-<date>.html` | Primary — self-contained, two-tab, opens in any browser |
| `output/audit-report-<date>.pdf` | Formal sharing — severity view only, print-ready |

The HTML file is fully self-contained: all CSS is inlined, all screenshots are embedded as base64 data URIs. No external dependencies at read time.

---

## HTML Structure

### Header
- RYZE wordmark (text, styled) + "QA Audit Report" + crawl date
- Summary stats bar: four colored count badges (Critical / High / Medium / Low)
- Tab switcher: **By Severity** | **By Category**

### Tab 1 — By Severity (default, also rendered in PDF)
Sections: Critical → High → Medium → Low. Each section has a colored heading and a ruled divider. Within each section, one card per unique finding, sorted by score descending.

### Tab 2 — By Category
Categories are determined dynamically during the orchestrate step (see below). Within each category, findings are sub-sorted by severity (Critical first), then score. Same card component as Tab 1.

### Finding Card
Each card contains, in order:
1. **Severity badge** — pill (color-coded: red / orange / amber / grey) + rule ID in monospace beside it
2. **Plain-English summary** — 1–2 sentences, generated by LLM (see Summary Generation below)
3. **Affected URLs** — listed as styled links; if more than 5, remainder is collapsed behind a "show N more" toggle
4. **Screenshot** — one per finding, cropped (see Screenshot Cropping below); caption shows viewport label

## Summary Generation

Summaries are generated during `scripts/orchestrate.ts` after scoring, before building the report. A summary is a 1–2 sentence plain-English explanation of the bug written for a non-technical reader.

| Severity | Model |
|---|---|
| Critical | claude-sonnet-4-6 |
| High | claude-sonnet-4-6 |
| Medium | claude-sonnet-4-6 |
| Low | claude-haiku-4-5-20251001 |

**Prompt (per finding):**
```
You are writing a plain-English bug summary for a non-technical stakeholder report.
Rule: {ruleId}
Description: {description}
Affected URLs: {urls joined, max 3}
Write 1–2 sentences that explain what is wrong and why it matters to a customer or the business.
Be specific — reference the actual content or URL if it helps. Do not use jargon.
Respond with only the summary text, no preamble.
```

Summaries are stored on `ScoredBug` as a new `summary: string` field. They are generated in parallel with `p-limit(10)` concurrency. Low findings use Haiku with the same prompt.

---

## Category Clustering

During orchestrate, after all summaries are generated, a single Haiku call reads the full list of finding descriptions and assigns each a short human-readable category label (e.g., "Sale Pricing", "Broken Links", "Missing Page Content", "Checkout Flow", "Blog Content", "SEO Tags"). The call returns a JSON map of `fingerprint → category`. Each `ScoredBug` gets a `category: string` field populated from this map.

**Prompt:**
```
You are categorizing QA findings for an e-commerce site report.
Below is a list of findings. Assign each a short category label (2–4 words) that describes
the type of problem. Use consistent labels across similar findings.
Return JSON only: { "<fingerprint>": "<category>", ... }

Findings:
{findings as JSON array with fingerprint, ruleId, description}
```

If the call fails, `category` falls back to a rule-prefix map:
- `revenue:*` → "Revenue & Checkout"
- `axe:*` → "Accessibility"
- `network:404` → "Broken Links"
- `seo:*` → "SEO Tags"
- `content:*` → "Content Quality"
- Everything else → "Other"

## Screenshot Cropping

Three-tier fallback, in priority order:

**Tier 1 — Element screenshot**
If `bug.elementShot` is populated, use it directly. This is captured during `scripts/reverify.ts` for any bug that has a `selector` field: Playwright navigates to the page, locates the element, calls `element.screenshot()`, saves to `output/screenshots/<fingerprint>-element.png`, and stores the path on the bug.

**Tier 2 — Sharp crop from full-page screenshot**
If no element screenshot exists but a full-page screenshot does, Sharp crops a 700×350px region. Crop anchor: top-left of the page (0, 0) for above-the-fold issues; for `network:404` and link bugs, use the upper half of the page since these are typically header/nav/body link failures.

**Tier 3 — Resized full-page fallback**
If neither is available, the full-page screenshot is resized to 700px wide and embedded. A grey label "Full page — element location unknown" appears below it.

All screenshots are embedded in the HTML as base64 `<img>` tags so the file is self-contained.

## PDF Export

Generated immediately after the HTML file by `src/report/pdf-exporter.ts` using Playwright's `page.pdf()`.

- Loads the HTML file in a headless browser page
- Injects a `?view=severity` query param which sets the severity tab as active and hides the tab switcher via CSS (`@media print` already handles this)
- Calls `page.pdf({ format: 'A4', printBackground: true, margin: { top: '1cm', bottom: '1cm', left: '1cm', right: '1cm' } })`
- The `@media print` stylesheet hides: tab switcher, "show more" toggles (all URLs expanded), interactive elements

## Type Changes

`ScoredBug` in `src/types.ts` gains two new optional fields:
```ts
summary?: string;   // LLM-generated plain-English summary
```

`BugRecord` is unchanged (summary and category live only on ScoredBug).

## Constraints

- HTML file must be fully self-contained (no CDN links, no external fonts — use system font stack)
- No new npm dependencies: Playwright is already installed for PDF; Sharp is already installed for image processing; Anthropic SDK is already installed for summaries
- Summary generation must not block on failure — if a Sonnet call fails, fall back to the raw `description` field truncated to 200 chars
- Category clustering must not block on failure — use rule-prefix fallback map
- PDF export must not fail the overall report build — if Playwright PDF fails, log a warning and continue; the HTML file is the primary deliverable
