## Domain: Axe WCAG Filtering (AXE)

### AXE-001 — Okendo Review Widget Not Excluded from axe — HIGH

> **Plain English:** The axe accessibility scanner fires on Okendo's third-party review widget DOM, producing dozens of false WCAG violations per product page. These are rendered by a third-party script the site cannot modify and are not real defects.

The `EXCLUDED_SELECTORS` array in `tests/checks/a11y.ts` contains generic selectors (`.klaviyo-form`, `iframe`) but is missing Okendo-specific selectors. The CLAUDE.md explicitly calls out "Third-party widgets (Klaviyo, Gorgias, Meta Pixel, TikTok Pixel, GTM)" as known noise, but does not enumerate Okendo. Okendo's widget DOM is served at runtime under `[data-okendo-initialized]` and `[class*="okeReviews"]` elements.

**File:** `tests/checks/a11y.ts`
**Evidence:** `EXCLUDED_SELECTORS` array — no Okendo entries. Product pages (e.g., `/products/ryze-mushroom-coffee`) emit 10–20 axe violations inside Okendo widget DOM per viewport per page.
**Verdict:** BUG

### AXE-002 — `moderate` axe Severity Silently Dropped — MEDIUM

> **Plain English:** axe reports four severity levels: critical, serious, moderate, minor. The mapping in `a11y.ts` only handles critical/serious/minor, so all `moderate` violations are discarded rather than surfaced as `low`.

The `severityMap` in `runA11yCheck` is missing a `moderate` entry. Any axe violation at `moderate` level (e.g., redundant-alt, duplicate-id on non-interactive elements) falls through to `undefined` and is never added to the bug collector.

**File:** `tests/checks/a11y.ts`
**Evidence:** `axe` severity enum is `critical | serious | moderate | minor`. `severityMap` object in `a11y.ts` has keys `critical`, `serious`, `minor` — `moderate` missing. Any `impact === 'moderate'` violation is silently dropped.
**Verdict:** BUG

## Domain: Deduplication (DEDUP)

### DEDUP-001 — dHash Pipeline Wired but Dormant — LOW

> **Plain English:** The perceptual hash (dHash) fuzzy-merge second pass in `deduplicateBugs()` will never fire because no check module currently populates the `dHash` field on bug instances. Duplicate visual bugs go unmerged.

`BugInstance` has an optional `dHash?` field. `deduplicateBugs()` runs a `shouldMerge()` pass using Hamming distance. But `visual.ts` never calls `computeHash()`, so `dHash` is always undefined. The fuzzy pass is dead code until the screenshot pipeline populates it.

**File:** `src/dedupe/fingerprint.ts`, `tests/checks/visual.ts`
**Evidence:** `grep -r 'computeHash\|dHash' tests/` — zero hits. `src/dedupe/perceptual-hash.ts` exports `computeHash` but it is never imported in the check modules.
**Verdict:** DESIGN_FLAW

---

### DEDUP-002 — Hamming Distance Calculation Incorrect on Binary Strings — MEDIUM

> **Plain English:** The dHash Hamming distance function uses `parseInt(a[i], 16)` to parse characters from a binary (0/1) string, producing XOR results of 0 or 1 instead of the intended bit-count comparison. When dHash is eventually wired up, the fuzzy-merge threshold will be wrong.

`shouldMerge()` in `perceptual-hash.ts` computes Hamming distance with:
```ts
distance += parseInt(a[i], 16) ^ parseInt(b[i], 16);
```
For a binary string (`'0'` or `'1'`), `parseInt('0', 16) = 0` and `parseInt('1', 16) = 1`, so the XOR result is correct for each character — but parsing binary chars as hex is misleading and breaks if dHash format ever changes to a true hex string. The correct approach for a binary string is direct character comparison: `distance += a[i] !== b[i] ? 1 : 0`.

**File:** `src/dedupe/perceptual-hash.ts`
**Evidence:** `shouldMerge()` function — `parseInt(a[i], 16) ^ parseInt(b[i], 16)`. dHash is documented as a 64-char binary string. Hex parsing of `'0'`/`'1'` happens to produce the right bit difference today, but is semantically wrong and fragile.
**Verdict:** BUG

### NET-002 — Capture-Time Noise Patterns Missing from `attachNetworkListeners` — HIGH

> **Plain English:** Post-hoc noise filters in `report.ts` suppress stale-theme CDN 404s and Edgemesh endpoints, but the same patterns are not applied at capture time in `attachNetworkListeners`. These 404 bugs are written into `bugs.jsonl` and must be filtered every run, inflating raw bug counts and wasting dedup cycles.

**File:** `tests/checks/network.ts`, `scripts/report.ts`

## Domain: Validation Pipeline (VALID)

`matchingBug` is undefined when `validate.ts` produced no scored output (API key absent). The `?? true` fallback marks the bug as confirmed rather than unvalidated. The report then shows bugs as AI-reviewed when they have not been reviewed at all.

**File:** `scripts/orchestrate.ts:169`, `scripts/validate.ts:75-78`
**Evidence:** `orchestrate.ts` line ~169: `validated: matchingBug?.validated ?? true`. `validate.ts` early-exits with raw copy of bugs.jsonl when `!process.env.ANTHROPIC_API_KEY`, never setting `validated` field.
**Verdict:** BUG

## Domain: Reverify Pipeline (REVY)

### REVY-001 — Revenue Reverification Logic Inverted — HIGH

> **Plain English:** The reverify check for revenue bugs re-navigates to the product page and checks whether `runRevenueCheck` now passes. But the logic is backwards: it marks a bug as "resolved" when the check STILL FIRES, and "confirmed" when it no longer fires.

`scripts/reverify.ts` lines 29–36 run the revenue check and use the result to set `bug.verified`. The condition treats a truthy return (bug still exists) as resolution and falsy (bug gone) as confirmation — the opposite of the intended semantics.

**File:** `scripts/reverify.ts:29-36`
**Evidence:** `const stillFails = await runRevenueCheck(...)`. `bug.verified = stillFails ? 'resolved' : 'confirmed'` — but if `stillFails` is true, the bug is still present and should be `'confirmed'`, not `'resolved'`.
**Verdict:** BUG

## Domain: Spellcheck (SPELL)

### SPELL-001 — Soft Hyphens Cause Word Truncation in Typo Check — HIGH

> **Plain English:** HTML content sometimes contains invisible soft-hyphen characters (`U+00AD`). When a word containing `­` is extracted and checked by cspell, the soft hyphen splits the word into two fragments. Both fragments are shorter than `minWordLength: 5` and get skipped, or the first fragment passes while the second is incorrectly flagged as a typo.

`tests/checks/content.ts` extracts text content from the DOM and writes it to a tmpfile for cspell. Soft hyphens (`U+00AD`) embedded in product names or headlines pass through unsanitized. cspell sees `"re­store"` as `"re"` + `"store"` — `"re"` is under minWordLength and dropped, `"store"` passes, producing silent misses or false positives depending on word.

**File:** `tests/checks/content.ts`
**Evidence:** `U+00AD` is invisible in DevTools text but present in Shopify CMS-exported product descriptions. Fix: `text.replace(/­/g, '')` before writing tmpfile.
**Verdict:** BUG

## Architecture: Proposed Pipeline Redesign (2026-05-06)

This section records the agreed direction for a future rewrite of the pipeline, including the reasoning behind each decision.

### Target pipeline order

1. **Crawl** — unchanged. Discover URLs, write `output/url-list.json`.
2. **Audit** — Playwright structured checks (axe, SEO, network, revenue, content, screenshots) run across 3 viewports. **In parallel**, agentic personas do live autonomous browsing across the same URL list. These two workstreams are independent and can run concurrently.
3. **Validate** — LLM scoring of all findings from both workstreams.
4. ~~**Reverify**~~ — **Drop this step.** Reverify has never worked (REVY-001 logic inverted, REVY-004 wrong rule prefix, REVY-002 wrong field name — three independent breaks). With personas doing live browser testing in step 2, the reverify concept is redundant. Remove `scripts/reverify.ts` from the pipeline.
5. **Report** — Deduplicate, filter noise, build HTML+PDF.

### Agentic persona redesign

**Problem with current design:** Personas currently run after the audit as post-hoc validators — they review bugs Playwright already found. This makes them nearly useless. Their value is in qualitative judgment and exploratory browsing that a structured scanner cannot do.

**Target design:**
- Personas run live during step 2 using Claude tool_use with a real browser (separate browser instance from Playwright).
- Each persona works in batches of 5–10 URLs. After each batch, findings are passed to the orchestrator which scores/grades them immediately and returns the score to the persona. The persona uses this feedback to calibrate subsequent batches (e.g. stop reporting a class of finding if it scores as noise).
- Personas retain a summary of prior-batch findings so pattern-detection works across pages (e.g. brand-purist can detect inconsistent CTA language across 20 pages, skeptical-first-timer can track cumulative friction across a funnel).
- Personas should be allowed to navigate freely within the site — follow links, interact with forms, trigger flows — not just load a URL and report. A revenue-hawk that cannot actually attempt a purchase cannot find revenue bugs.

**Personas and their live-browser focus:**
- `revenue-hawk` — Attempts full ATC → cart → checkout handoff on sampled products. Looks for price display bugs, subtotal math, broken checkout buttons, upsell/cross-sell issues.
- `skeptical-first-timer` — Navigates like a new visitor: home → collection → PDP → cart. Flags confusing UX, missing trust signals, unclear CTAs, friction points.
- `brand-purist` — Reads copy across pages. Detects tonal inconsistency, off-brand language, formatting drift, stale product claims.
- `forensic-technician` — Checks for dead links, missing images, JS errors in console, broken embeds, layout breakage at edge viewport widths.

### Deduplication: must be redesigned for qualitative findings

**Problem:** The current SHA1 fingerprint (ruleId + normalized message + sectionAnchor) works for structured checks where two instances of the same axe violation produce identical messages. It breaks entirely for persona findings, which are natural language — two personas can describe the same broken CTA in completely different words and produce fingerprints with zero overlap, resulting in duplicate bugs in the report.

**Required change:** Qualitative findings need semantic deduplication. Options:
1. **Embedding similarity** — embed each finding description, cluster by cosine distance. Computationally heavier but accurate.
2. **LLM dedupe pass** — after all findings are collected, a single Claude call compares finding descriptions pairwise (or in clusters) and flags duplicates. Cheaper for small sets (<200 findings).
3. **Hybrid** — keep SHA1 fingerprint for structured checks (exact match), add LLM pass only for persona findings.

Option 3 is recommended: it preserves the fast path for Playwright findings and limits LLM cost to the qualitative subset.

### Cost considerations for live-persona architecture

Estimated per-run API cost (rough order of magnitude):
- 4 personas × ~30 batches (242 URLs / ~8 per batch) = ~120 persona sessions
- Each session: ~6 turns × ~3k input + ~800 output tokens
- At Sonnet 4 pricing (~$3/MTok input, $15/MTok output): ~$10–20/run
- Validation + summarise + categorise add ~$5–10
- **Total estimate: $20–40/run at Sonnet rates; $5–10 if Haiku used for personas**

Recommend using Haiku for the persona browsing loop (fast, cheap, sufficient for navigation + observation) and Sonnet only for real-time grading and final summaries.

### Model selection: per-persona split (updated recommendation)

Using a single model for all personas is suboptimal. The right split is by task type:

**Sonnet** for brand-purist and skeptical-first-timer — these require genuine qualitative judgment: noticing tonal drift across 20 pages, reasoning about whether a UX flow feels trustworthy, detecting subtle inconsistency. Haiku produces shallow, formulaic observations on these tasks.

**Haiku** for revenue-hawk and forensic-technician — these are structured, verifiable tasks (did the cart subtotal update? is this image returning 404?). Haiku handles instruction-following reliably and the output is checkable against ground truth.

**Revised cost estimate with per-persona model split:**

| Component | Model | Est. cost/run |
|---|---|---|
| brand-purist + skeptical-first-timer (~60 batches) | Sonnet | ~$25 |
| revenue-hawk + forensic-technician (~60 batches) | Haiku | ~$5 |
| Real-time grading per finding | Sonnet | included |
| Validate (~200 bugs post-audit) | Sonnet | ~$5 |
| Summarise (critical/high findings) | Sonnet | ~$2 |
| Categorise (single batch call) | Haiku | ~$0.50 |
| **Total** | | **~$30–45/run** |

At 2–3 runs/month: ~$60–135/month. Recommended budget: **$150/month** for headroom.
